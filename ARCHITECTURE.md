# Последовательная работа соединения в `port-tunnel`

С точки зрения использования программа работает следующим образом.

Сначала на публичном сервере запускается серверная часть `port-tunnel`. У сервера есть публичный IP-адрес, поэтому к нему могут подключаться как клиенты туннеля, так и внешние пользователи. После запуска сервер начинает ждать подключения клиентов на управляющем порту, например `7000`.

> **Сноска 1. Что происходит в коде**
>
> Сервер создаёт TCP-listener на управляющем порту:
>
> ```python
> control_server = await asyncio.start_server(
>     handle_control_or_data,
>     host="0.0.0.0",
>     port=7000,
> )
> ```
>
> Это означает, что все новые TCP-соединения на порт `7000` будут передаваться обработчику `handle_control_or_data`.

Затем на компьютере пользователя запускается локальный сервис. Это может быть сайт, API, игровой сервер или любая другая программа, принимающая TCP-подключения. Например, локальный сайт может работать на `127.0.0.1:8080`.

Сам по себе такой сервис доступен только с компьютера пользователя или из его локальной сети.

> **Сноска 2. Что такое локальный сервис для программы**
>
> Для клиента туннеля локальный сервис определяется двумя параметрами:
>
> ```python
> local_host = "127.0.0.1"
> local_port = 8080
> ```
>
> Позже клиент будет подключаться к этому адресу как обычный TCP-клиент.

После этого пользователь запускает клиентскую часть `port-tunnel` и указывает:

* адрес публичного сервера;
* управляющий порт сервера;
* адрес и порт локального сервиса;
* публичный порт, через который сервис должен быть доступен извне.

Например:

```text
сервер туннеля: VPS_IP:7000
локальный сервис: 127.0.0.1:8080
публичный порт: 30001
```

Клиент самостоятельно подключается к публичному серверу. Это важно, потому что компьютер клиента может находиться за NAT. Исходящие соединения обычно разрешены маршрутизатором, поэтому никаких настроек проброса портов на домашнем роутере не требуется.

> **Сноска 3. Создание control-соединения**
>
> Клиент открывает исходящее TCP-соединение:
>
> ```python
> control_reader, control_writer = await asyncio.open_connection(
>     server_host,
>     control_port,
> )
> ```
>
> Это постоянное управляющее соединение между клиентом и сервером.

После подключения клиент сообщает серверу, что хочет создать туннель.

> **Сноска 4. Регистрация туннеля**
>
> Клиент отправляет служебное сообщение:
>
> ```python
> await send_json(control_writer, {
>     "type": "register",
>     "public_port": 30001,
> })
> ```
>
> Сервер по первому сообщению понимает, что это регистрация нового туннеля.

Сервер регистрирует туннель и начинает принимать внешние подключения на указанном публичном порту. В результате появляется такое соответствие:

```text
VPS_IP:30001 → локальный сервис 127.0.0.1:8080
```

При этом прямого соединения между публичным сервером и локальным сервисом ещё нет. Сервер лишь знает, какой клиент отвечает за этот публичный порт, и поддерживает с ним постоянное управляющее соединение.

> **Сноска 5. Создание публичного listener**
>
> Сервер запускает отдельный listener для конкретного туннеля:
>
> ```python
> public_server = await asyncio.start_server(
>     handle_public_connection,
>     host="0.0.0.0",
>     port=30001,
> )
> ```
>
> Также сервер сохраняет:
>
> ```python
> tunnel = {
>     "tunnel_id": tunnel_id,
>     "public_port": 30001,
>     "control_writer": control_writer,
> }
> ```

Далее внешний пользователь открывает публичный адрес, например:

```text
http://VPS_IP:30001
```

Его браузер устанавливает TCP-соединение с сервером туннеля. Сервер принимает это подключение, временно сохраняет его и сообщает клиентской программе, что появился новый внешний пользователь.

> **Сноска 6. Внешнее подключение**
>
> При новом подключении сервер получает:
>
> ```python
> public_reader
> public_writer
> ```
>
> Это две стороны одного внешнего TCP-соединения:
>
> * `public_reader` читает данные от внешнего пользователя;
> * `public_writer` отправляет данные внешнему пользователю.
>
> Сервер создаёт идентификатор:
>
> ```python
> connection_id = generate_connection_id()
> ```
>
> и временно сохраняет соединение:
>
> ```python
> pending_connections[connection_id] = (
>     public_reader,
>     public_writer,
> )
> ```

После этого сервер уведомляет клиента по управляющему соединению.

> **Сноска 7. Уведомление клиента**
>
> Сервер отправляет:
>
> ```python
> await send_json(control_writer, {
>     "type": "new_connection",
>     "tunnel_id": tunnel_id,
>     "connection_id": connection_id,
> })
> ```
>
> Это сообщение означает: «На публичный порт пришёл новый пользователь. Открой для него отдельный data-канал».

Получив уведомление, клиентская программа создаёт новое отдельное соединение с сервером. Это уже не управляющее соединение, а канал передачи данных для конкретного внешнего пользователя.

> **Сноска 8. Создание data-соединения**
>
> Клиент повторно подключается к тому же серверному порту:
>
> ```python
> data_reader, data_writer = await asyncio.open_connection(
>     server_host,
>     control_port,
> )
> ```
>
> Затем идентифицирует это соединение:
>
> ```python
> await send_json(data_writer, {
>     "type": "data",
>     "tunnel_id": tunnel_id,
>     "connection_id": connection_id,
> })
> ```
>
> После этого данный TCP-канал используется уже не для команд, а для передачи пользовательского трафика.

Одновременно клиент подключается к локальному сервису, например к `127.0.0.1:8080`.

> **Сноска 9. Подключение к локальному сервису**
>
> Клиент создаёт ещё одно TCP-соединение:
>
> ```python
> local_reader, local_writer = await asyncio.open_connection(
>     local_host,
>     local_port,
> )
> ```
>
> Теперь у клиента есть:
>
> ```text
> data-соединение с VPS
> локальное соединение с сервисом
> ```

В этот момент формируется полная цепочка:

```text
внешний пользователь
→ публичный сервер
→ клиент туннеля
→ локальный сервис
```

Сервер связывает внешнее подключение пользователя с новым каналом клиента. Клиент, в свою очередь, связывает этот канал с локальным сервисом.

## Какие мосты создаются

После этого в системе существуют два независимых двунаправленных моста.

### Мост на сервере

Сервер связывает:

```text
внешнее TCP-соединение
↔
data-соединение клиента
```

Упрощённо это соответствует вызову:

```python
await bridge(
    public_connection.reader,
    public_connection.writer,
    data_reader,
    data_writer,
)
```

В реальном серверном обработчике аргументы:

```python
bridge(
    public_connection.reader,
    public_connection.writer,
    reader,
    writer,
)
```

означают:

```text
public_connection.reader
    читает данные от внешнего пользователя

public_connection.writer
    отправляет данные внешнему пользователю

reader
    читает данные из data-соединения клиента

writer
    отправляет данные в data-соединение клиента
```

Внутри этого моста запускаются два направления передачи:

```python
asyncio.create_task(
    pipe(public_connection.reader, writer)
)
```

Это направление:

```text
внешний пользователь
→ сервер
→ data-соединение
→ клиент
```

И второе:

```python
asyncio.create_task(
    pipe(reader, public_connection.writer)
)
```

Это направление:

```text
клиент
→ data-соединение
→ сервер
→ внешний пользователь
```

Итоговый серверный мост:

```text
public socket ↔ data socket
```

### Мост на клиенте

Клиент связывает:

```text
data-соединение с сервером
↔
соединение с локальным сервисом
```

Упрощённо:

```python
await bridge(
    data_reader,
    data_writer,
    local_reader,
    local_writer,
)
```

Внутри создаются два направления.

Первое:

```python
asyncio.create_task(
    pipe(data_reader, local_writer)
)
```

Передаёт:

```text
сервер
→ клиент
→ локальный сервис
```

Второе:

```python
asyncio.create_task(
    pipe(local_reader, data_writer)
)
```

Передаёт:

```text
локальный сервис
→ клиент
→ сервер
```

Итоговый клиентский мост:

```text
data socket ↔ local socket
```

Оба моста вместе образуют непрерывный логический канал:

```text
внешний пользователь
        ↕
серверный public socket
        ↕
серверный data socket
        ↕
клиентский data socket
        ↕
клиентский local socket
        ↕
локальный сервис
```

При этом:

* `серверный data socket`;
* `клиентский data socket`

являются двумя сторонами одного TCP-соединения между клиентом и VPS.

Именно поэтому полная цепочка фактически выглядит так:

```text
[внешний пользователь]
        ↕
[Мост №1 на сервере]
        ↕
[data TCP-соединение]
        ↕
[Мост №2 на клиенте]
        ↕
[локальный сервис]
```

После формирования мостов начинается обычная передача данных.

Когда браузер отправляет HTTP-запрос, он проходит:

```text
браузер
→ публичный порт VPS
→ серверный мост
→ data-соединение
→ клиентский мост
→ локальный HTTP-сервер
```

> **Сноска 10. Передача запроса**
>
> Сервер и клиент не анализируют HTTP-запрос. Они просто читают очередную порцию байтов и передают дальше:
>
> ```python
> data = await reader.read(64 * 1024)
> writer.write(data)
> await writer.drain()
> ```

Локальный сервис обрабатывает запрос и формирует ответ. Ответ проходит в обратном направлении:

```text
локальный HTTP-сервер
→ клиентский мост
→ data-соединение
→ серверный мост
→ браузер
```

Для внешнего пользователя всё выглядит так, будто сервис действительно запущен на VPS. Он не знает, что фактически сайт или игровой сервер работает на другом компьютере за NAT.

Управляющее соединение при этом продолжает существовать всё время работы клиента. Оно используется сервером, чтобы сообщать клиенту о новых внешних подключениях.

Для каждого нового внешнего подключения создаётся отдельный канал передачи данных. Поэтому несколько пользователей могут работать с одним туннелем одновременно.

Например, если два браузера одновременно открывают публичный адрес, схема будет такой:

```text
пользователь 1
→ public connection 1
→ data connection 1
→ локальный сервис

пользователь 2
→ public connection 2
→ data connection 2
→ локальный сервис
```

> **Сноска 11. Параллельная обработка**
>
> Для каждого нового соединения клиент запускает отдельную задачу:
>
> ```python
> asyncio.create_task(
>     handle_new_connection(...)
> )
> ```
>
> Благодаря этому несколько соединений обслуживаются одновременно.

Если внешний пользователь закрывает страницу или завершает соединение, соответствующий канал передачи данных закрывается. Сам туннель продолжает работать и может принимать следующие подключения.

> **Сноска 12. Завершение отдельного соединения**
>
> Когда одна сторона перестаёт передавать данные:
>
> ```python
> data = await reader.read(...)
> if not data:
>     break
> ```
>
> мост завершает связанные задачи и закрывает сокеты.

Если клиентская программа полностью останавливается или теряет соединение с сервером, управляющее соединение разрывается. Сервер понимает, что туннель больше недоступен, перестаёт обслуживать соответствующий публичный порт и освобождает связанные ресурсы.

> **Сноска 13. Удаление туннеля**
>
> Упрощённо:
>
> ```python
> await unregister_tunnel(tunnel_id)
> public_server.close()
> await public_server.wait_closed()
> ```
>
> После этого публичный порт освобождается.

Таким образом, полный жизненный цикл выглядит так:

1. Запускается публичный сервер туннеля.
2. Запускается локальный сервис.
3. Запускается клиент туннеля.
4. Клиент устанавливает исходящее управляющее соединение с сервером.
5. Сервер регистрирует туннель и открывает публичный порт.
6. Внешний пользователь подключается к публичному порту.
7. Сервер уведомляет клиента о новом подключении.
8. Клиент создаёт отдельный канал передачи данных.
9. Клиент подключается к локальному сервису.
10. На сервере создаётся мост `public socket ↔ data socket`.
11. На клиенте создаётся мост `data socket ↔ local socket`.
12. Данные передаются в обе стороны.
13. После ухода пользователя его отдельное соединение закрывается.
14. Туннель продолжает работать до остановки клиента.
