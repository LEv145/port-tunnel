# Архитектура Port Tunnel

## Общая схема

```text
Внешний пользователь
        |
        | TCP к public_port
        v
TCPTunnelServer
        |
        | отдельное data-соединение
        v
TCPTunnelClient
        |
        | TCP к local_host:local_port
        v
Локальное приложение
```

Все соединения между клиентом и сервером открываются клиентом, поэтому клиент может находиться за NAT.

## Основные компоненты

### `ControlMessageCodec`

Кодирует управляющее сообщение в формат:

```text
[4 байта длины][JSON UTF-8]
```

Размер сообщения ограничивается параметром `max_message_size`.

### `ControlChannel`

Владеет `StreamReader` и `StreamWriter` управляющего соединения.

Он предоставляет:

- `send()` — последовательную отправку типизированных сообщений;
- `receive()` — чтение одного сообщения;
- `close()` — закрытие канала;
- `detach()` — передачу потоков в raw TCP-мост после `DataMessage`.

Одновременное чтение одного канала запрещено. Параллельная отправка сериализуется через `asyncio.Lock`.

### `TCPTunnelClient`

Клиент:

1. подключается к control-порту сервера;
2. отправляет `RegisterMessage`;
3. получает `RegisteredMessage`;
4. читает `NewConnectionMessage` и `PingMessage`;
5. создаёт отдельное data-соединение для каждого внешнего подключения;
6. отвечает на `PingMessage` сообщением `PongMessage`;
7. связывает data-соединение с локальным TCP-сервисом.

### `TCPTunnelServer`

Сервер:

1. принимает `RegisterMessage` и `DataMessage` на одном control-порту;
2. проверяет клиентский токен;
3. открывает отдельный публичный listener;
4. хранит туннель в `TCPTunnelRegistry`;
5. уведомляет клиента через `NewConnectionMessage`;
6. связывает внешний сокет с data-соединением;
7. запускает heartbeat;
8. удаляет туннель при отключении или heartbeat timeout.

## Управляющий протокол

Текущие сообщения:

- `RegisterMessage`;
- `RegisteredMessage`;
- `NewConnectionMessage`;
- `DataMessage`;
- `ErrorMessage`;
- `PingMessage`;
- `PongMessage`.

Сообщения описаны Pydantic-моделями и выбираются по полю `type`.

## Heartbeat

Сервер является инициатором heartbeat.

```text
сервер -> PingMessage(heartbeat_id)
клиент -> PongMessage(heartbeat_id)
```

После успешного ответа сервер ждёт `heartbeat_interval` и отправляет следующий `ping`.

Если соответствующий `pong` не получен за `heartbeat_timeout`, сервер:

1. закрывает управляющий канал;
2. прекращает принимать подключения на публичном порту;
3. удаляет туннель из реестра;
4. закрывает ожидающие подключения;
5. закрывает активные TCP-мосты;
6. освобождает публичный порт.

Только основной control-loop читает сообщения. Heartbeat-задача только отправляет `ping` и ожидает событие,
которое устанавливается control-loop после получения подходящего `pong`.

## Data-соединение

После внешнего подключения сервер создаёт `connection_id` и отправляет клиенту `NewConnectionMessage`.

Клиент открывает новое соединение к control-порту и первым кадром отправляет `DataMessage`. После этого обе стороны
вызывают `ControlChannel.detach()`, и оставшийся поток используется как обычный raw TCP-канал.

Создаются два двунаправленных моста:

```text
сервер: внешний пользователь <-> data-соединение
клиент: data-соединение <-> локальное приложение
```

## Завершение туннеля

Туннель удаляется при:

- нормальном завершении клиента;
- обрыве control-соединения;
- ошибке управляющего протокола;
- heartbeat timeout;
- отмене серверной задачи.

При удалении закрываются listener, control-канал, pending-соединения и active-соединения.
